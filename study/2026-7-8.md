# 如何实现窗口的右键菜单

推荐在窗口管理器中统一实现，使用系统Menu有更好的兼容性
```ts
// 监听右键菜单事件
win.webContents.on('context-menu', (event, params) => {
  // 实例化一个新菜单
  const menu = new Menu();
  menu.append(
    new MenuItem({
      label: '返回上一级',
      enabled: win.webContents.navigationHistory.canGoBack(),
      click: () => {
        win.webContents.navigationHistory.goBack();
      },
    }),
  );
  menu.append(new MenuItem({ type: 'separator' }));
  menu.append(
    new MenuItem({
      label: '刷新',
      click: () => {
        win.webContents.reload();
      },
    }),
  );
  menu.append(
    new MenuItem({
      label: '强制刷新',
      click: () => {
        win.webContents.reloadIgnoringCache();
      },
    }),
  );
  menu.popup({ window: win });
});
```

## 大厂常规的打包和更新思路是怎么样的

大厂在 Electron 桌面端的打包和更新思路上，核心追求是“安全合规、极速交付、无感更新、安全兜底”。

以下是行业标准的常规落地思路：

### 一、 工业级打包规范 (Packaging)

大厂基本抛弃了手动的本地打包，全面接入自动化流水线。

1. **主流工具链：** `electron-builder` 是绝对的主流，它能完美处理打包、生成安装包、增量更新清单（blockmap）等全套流程。
2. **源码保护 (ASAR 加密)：** * 基础：绝对开启 `asar` 打包，防止用户直接看到 `node_modules` 和明文源码。
* 进阶：对于核心业务，会在打包前使用 `javascript-obfuscator` 进行混淆，甚至修改 Electron 源码实现自定义的 asar 解密算法，防止竞品解包破解。


3. **数字签名与公证 (必须具备)：**
* **Windows：** 购买 EV 代码签名证书（需实体 U 盾或云签名），消除 Windows Defender 报毒和“未知发布者”拦截。
* **macOS：** 使用 Apple Developer 证书签名，并**必须接入 Notarization（公证）流程**，否则在 macOS 10.15+ 上直接无法运行。


4. **CI/CD 自动化构建：** * 绝不在开发者的个人电脑上打包。通过 GitLab CI、GitHub Actions 或 Jenkins 配合 Mac/Windows/Linux 专属构建集群，实现一键打出多端安装包。

---

### 二、 分层更新策略 (Updating)

大厂绝不会只依赖一种更新方式，而是采用“双轨制”：

#### 1. 业务线主力：纯前端“热更新” (Hot Update)

* **适用场景：** 修改 UI、修复纯前端 Bug、增加常规业务逻辑（不涉及 C++ 原生模块或 Electron 版本升级）。
* **核心思路：** 绕过繁重的系统安装过程。
* **实现方式：** 1. 将前端打包产物（或整个 `app.asar`）放在独立的服务器（或 CDN）上。
2. 每次启动或后台定时去请求版本接口。
3. 发现新版本后，静默下载 zip 包并解压到应用的用户数据目录 (`app.getPath('userData')`)。
4. 下次启动窗口时，动态修改加载路径，让 Electron 读取新目录下的 `index.html` 或 `asar` 文件。
* **优势：** 极快，完全无感，不需要管理员权限，不需要重启整个底层框架。

#### 2. 底层架构兜底：全量/增量安装包更新 (App Update)

* **适用场景：** 升级 Electron 内核、升级 Node.js、引入了新的 C++ 动态链接库 (.node)、热更新出现致命错误。
* **核心思路：** 替换整个应用的基础运行环境。
* **实现方式：** 依赖 `electron-updater`。
* **增量更新 (Delta)：** 比较新旧版本的 blockmap，只下载变动的二进制分块。比如 100MB 的包可能只需要下载 5MB。
* **体验优化：** 后台静默下载。下载完成后，弹窗提示用户“新版本已准备就绪，是否重启安装？”。



---

### 三、 质量管控：灰度与回滚机制

大厂绝不会把新版本一次性推给所有用户，这是引发灾难的温床。

1. **白名单与灰度发布 (Grayscale Release)：**
* 更新接口不会直接返回最新版，而是根据当前用户的 ID 或设备指纹计算 Hash 命中率。
* **节奏：** 内部测试名单 -> 1% 种子用户 -> 10% 活跃用户 -> 50% -> 100% 全量。一旦监控系统（如 Sentry）发现崩溃率飙升，立刻停止放量。


2. **容灾回滚机制 (Rollback)：**
* 如果热更新拉取的新代码导致了白屏，应用必须具备**自愈能力**。
* 通常会在启动时检测主进程是否崩溃、渲染进程是否卡死（即我们之前讨论的各种监控）。如果连续崩溃 3 次，自动清空用户数据目录下的热更新补丁，回退到安装包自带的“出厂版本”，并向服务端上报异常。
