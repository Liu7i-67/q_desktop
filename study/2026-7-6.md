# AI说

在大厂（如飞书、VS Code、Slack、Notion 等）的桌面客户端开发中，由于涉及极高的**性能要求（降低内存和 CPU 占用）**、**严格的安全合规**以及**复杂的多窗口协同**，这三个全局事件通常扮演着“中央调度枢纽”的角色。

以下是大厂在这些事件中的典型工程实践：

## 1. `browser-window-created`：中枢注册与安全防御

大厂极少放任业务侧随意创建裸窗口，所有窗口诞生时都会经过这个“安检站”。

- **注入窗口管理器（Window Manager）**：
  将新生成的窗口实例（及其 `id`）推入全局的“窗口池（Window Pool）”。飞书或 Slack 中有大量独立窗口（主界面、独立会话、日历、会议、图片查看器），通过全局管理，可以实现“关闭主窗口时，联动关闭所有子窗口”或“群发广播消息”。
- **挂载性能探针（APM/Telemetry）**：
  打点记录窗口创建的绝对时间戳。后续配合渲染进程的首屏加载完成事件，计算出 `白屏时间` 和 `FMP（首次有效绘制）`，上报给数据后台进行性能监控。
- **统一的底层安全兜底**：
  即便在 `webPreferences` 里做了限制，大厂依然会在此时对 `webContents` 施加二次拦截：
- 统一接管 `will-navigate`，利用正则白名单拦截任何未知的外部跳转。
- 统一拦截 `select-bluetooth-device` 或 `media-paused` 等系统级权限请求。

## 2. `browser-window-focus`：资源满血与交互接管

当应用回到前台，大厂应用会立刻进入“备战状态”，接管系统资源并保证体验的实时性。

- **注册应用级快捷键（且避免冲突）**：
  这是最核心的操作。大厂绝不会在 `app.whenReady` 中直接注册 `globalShortcut`（因为这会让用户在使用其他软件时快捷键失灵，引发严重投诉）。他们会在 `focus` 时注册全局快捷键（如搜索框唤起、截图），确保只在应用活跃时生效。
- **解除性能节流（Unthrottling）**：
  通知渲染进程：“现在我是前台了，全速运行”。恢复 WebSocket 的高频心跳，恢复 GSAP 或 Lottie 的复杂动画，或者恢复被挂起的长轮询网络请求。
- **剪贴板嗅探与智能提示**：
  在办公软件（飞书/钉钉）中，当窗口获得焦点时，主进程会静默读取系统剪贴板。如果发现剪贴板里是一张刚刚截取的图片或一个特定的会议链接，界面上会立刻弹出一个“是否要发送此图片/加入会议”的悬浮提示。
- **清理系统通知与角标**：
  清空 macOS 的 Dock 栏红点或 Windows 任务栏的闪烁状态（`app.setBadgeCount(0)`），并向系统级通知中心发送指令，清除与该应用相关的历史通知。

## 3. `browser-window-blur`：降级省电与现场保护

当应用退居后台，大厂的首要目标是**降低系统占用（特别是省电）**和**保护用户数据**。

- **注销快捷键**：
  精准调用 `globalShortcut.unregisterAll()`（或注销特定的一组快捷键），将系统控制权还给操作系统。
- **触发深度节流与资源释放**：
  这是大型 Electron 应用优化内存的关键手段。
- 通过 IPC 通知渲染进程，冻结非必要的 React/Vue 组件更新。
- 降低或停止某些后台轮询（例如从每 5 秒拉取一次消息变为每 30 秒一次）。
- 如果是一个包含 3D 渲染的界面，会在此时大幅降低 Canvas 的渲染帧率，减少 GPU 功耗。

- **防数据丢失（Auto-Save）**：
  当焦点离开，通常意味着用户的核心输入流被打断。此时会静默触发草稿箱的保存、将关键日志写入本地 IndexedDB 或 SQLite，确保即使用户直接在任务管理器里杀掉进程，数据也不会丢失。
- **隐私遮罩（企业安全要求）**：
  对于企业财务软件或强保密级通讯工具，失去焦点时，主进程会通知前端立刻在整个 `<body>` 上盖一层高斯模糊遮罩。这可以防止用户在会议投屏或背后有人走动时，不小心暴露敏感信息。
